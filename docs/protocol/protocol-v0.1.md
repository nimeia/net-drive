# Developer Mount Protocol v0.1

## 1. Scope

Protocol v0.1 defines the control-plane baseline for a Windows-first, editor-optimized remote filesystem.

This revision freezes:
- wire framing
- common identifiers
- header fields
- control-channel message schema
- error model
- session lifecycle expectations

This revision does **not** yet implement:
- metadata channel operations
- data channel operations
- watcher streaming
- recovery replay
- file locking semantics

## 2. Transport

### 2.1 Current implementation

The current Iter 1 code uses a **TCP placeholder transport**.

Wire format:
1. 4-byte big-endian frame length
2. fixed-size binary header
3. payload bytes

Payload encoding in Iter 1:
- UTF-8 JSON

### 2.2 Planned evolution

The framing layout is intentionally transport-agnostic so the same header and payload model can later be carried by:
- QUIC bidirectional streams
- QUIC datagrams for selected event traffic
- TCP/TLS during bring-up or fallback

## 3. Frame Header

The header is always 32 bytes.

| Offset | Size | Field | Type | Description |
|---|---:|---|---|---|
| 0 | 4 | magic | bytes | ASCII `DMNT` |
| 4 | 1 | version | uint8 | protocol version, current `1` |
| 5 | 1 | header_length | uint8 | bytes in header, current `32` |
| 6 | 1 | channel | uint8 | logical channel id |
| 7 | 1 | opcode | uint8 | message opcode |
| 8 | 4 | flags | uint32 | request/response/error bits |
| 12 | 8 | request_id | uint64 | correlates request/response |
| 20 | 8 | session_id | uint64 | `0` until a session exists |
| 28 | 4 | payload_length | uint32 | number of payload bytes |

### 3.1 Channel values

| Value | Name | Purpose |
|---:|---|---|
| 1 | control | handshake, auth, session, heartbeat |
| 2 | metadata | lookup, getattr, readdir, open |
| 3 | data | read, write, flush, close |
| 4 | events | watch subscribe, event stream, ack |
| 5 | recovery | resume, handle recovery, revalidate |

### 3.2 Flags

Flags are bitmasks.

| Bit | Constant | Meaning |
|---:|---|---|
| 0 | `FlagRequest` | frame is a request |
| 1 | `FlagResponse` | frame is a response |
| 2 | `FlagError` | response carries error |
| 3 | `FlagAckRequired` | sender expects explicit ack |
| 4 | `FlagReplay` | frame is replayed during recovery |
| 5 | `FlagCompressed` | payload is compressed |

Valid Iter 1 combinations:
- request: `FlagRequest`
- success response: `FlagResponse`
- error response: `FlagResponse | FlagError`

## 4. Common Types

### 4.1 SessionId
- type: `uint64`
- generated by server
- `0` means no session bound

### 4.2 RequestId
- type: `uint64`
- generated by client per in-flight request
- unique within the active client connection

### 4.3 Timestamp
- type: RFC3339 string in JSON payloads for v0.1
- planned binary encoding later

### 4.4 ErrorCode
- type: string enum in payload
- mapped to internal numeric categories in code

### 4.5 CapabilitySet

```json
{
  "transport": ["tcp-json"],
  "channels": ["control"],
  "features": ["auth-basic", "session-create", "heartbeat"]
}
```

## 6. Message Schemas

### 6.1 HelloReq

Direction: client -> server

```json
{
  "client_name": "devmount-client",
  "client_version": "0.1.0",
  "supported_protocol_versions": [1],
  "capabilities": {
    "transport": ["tcp-json"],
    "channels": ["control"],
    "features": ["auth-basic", "session-create", "heartbeat"]
  }
}
```

### 6.2 HelloResp

Direction: server -> client

```json
{
  "server_name": "devmount-server",
  "server_version": "0.1.0",
  "selected_protocol_version": 1,
  "server_time": "2026-03-14T12:00:00Z",
  "capabilities": {
    "transport": ["tcp-json"],
    "channels": ["control"],
    "features": ["auth-basic", "session-create", "heartbeat"]
  }
}
```

### 6.3 AuthReq

Direction: client -> server

```json
{
  "scheme": "dev-token",
  "token": "devmount-dev-token"
}
```

### 6.4 AuthResp

Direction: server -> client

```json
{
  "authenticated": true,
  "principal_id": "developer",
  "display_name": "Developer",
  "granted_features": ["session-create", "heartbeat"]
}
```

### 6.5 CreateSessionReq

Direction: client -> server

```json
{
  "client_instance_id": "client-1",
  "requested_lease_seconds": 30,
  "mount_name": "workspace-dev"
}
```

### 6.6 CreateSessionResp

Direction: server -> client

```json
{
  "session_id": 1001,
  "lease_seconds": 30,
  "expires_at": "2026-03-14T12:00:30Z",
  "state": "active"
}
```

### 6.7 ResumeSessionReq

Direction: client -> server

```json
{
  "session_id": 1001,
  "client_instance_id": "client-1",
  "last_known_server_time": "2026-03-14T12:00:20Z"
}
```

Status in Iter 1:
- schema is frozen
- server returns `ERR_UNSUPPORTED_OPERATION`

### 6.8 HeartbeatReq

Direction: client -> server

```json
{
  "session_id": 1001,
  "client_time": "2026-03-14T12:00:10Z"
}
```

### 6.9 HeartbeatResp

Direction: server -> client

```json
{
  "session_id": 1001,
  "server_time": "2026-03-14T12:00:10Z",
  "expires_at": "2026-03-14T12:00:40Z",
  "state": "active"
}
```

### 6.10 ErrorResp payload

Direction: server -> client

```json
{
  "code": "ERR_AUTH_REQUIRED",
  "message": "authentication required",
  "retryable": false,
  "details": {
    "operation": "CreateSession"
  }
}
```

## 7. Error Codes

| Code | Meaning |
|---|---|
| ERR_INVALID_REQUEST | malformed payload or invalid field set |
| ERR_UNSUPPORTED_VERSION | no compatible protocol version |
| ERR_UNSUPPORTED_OPERATION | opcode/schema known but not implemented |
| ERR_AUTH_REQUIRED | authentication missing or invalid |
| ERR_SESSION_EXPIRED | session timed out |
| ERR_SESSION_NOT_FOUND | requested session unknown |
| ERR_CAPABILITY_MISMATCH | requested feature not available |
| ERR_INTERNAL | internal server error |

## 8. Session Rules

1. Client must send `HelloReq` first.
2. Until Hello succeeds, only Hello is legal.
3. After Hello succeeds, client may send `AuthReq`.
4. After Auth succeeds, client may send `CreateSessionReq`.
5. After session creation, client must send `HeartbeatReq` before lease expiry.
6. If heartbeat is not received before expiry, session enters `expired`.
7. ResumeSession is reserved but not implemented in Iter 1.
